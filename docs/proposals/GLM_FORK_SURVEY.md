# Swarm Core & Control Plane Survey

The fork has successfully consolidated a complex multi-agent topology into a durable control-plane, but the migration from legacy pub/sub has left deep, cross-cutting lock contention in the swarm fan-out paths and severe parameter-threading across server handlers. Specifically, the core fan-out loops incur massive lock thrash by sequentially awaiting asynchronous operations while holding overlapping state locks, and the server functions pass around 12+ parameters to manually wire up the global swarm state.

The highest-value work is moving the server's global swarm state into a shared struct to eliminate 400+ lines of boilerplate, and fixing the `async-hold-sync-lock` violations which currently serialize the entire daemon under heavy broadcast loads. Secondary wins come from optimizing the W1/W2 event-log truncation paths to avoid O(N) `contains` scans, cleaning up lifecycle orphan monitors, and bridging the MCP server to handle streaming tool outputs natively.

## 1. Create a `SwarmContext` struct to eliminate state parameter threading
**File:** `crates/jcode-app-core/src/server/comm_plan.rs`, `client_comm_message.rs`
**Description:** Every swarm communication handler (e.g., `handle_comm_propose_plan`, `handle_comm_message`) threads 10-15 arguments just to pass the various components of the global swarm state (`swarm_members`, `swarms_by_id`, `swarm_plans`, `swarm_coordinators`, plus queues and event history). Consolidate these into a single struct:
```rust
#[derive(Clone)]
pub(super) struct SwarmContext {
    pub members: Arc<RwLock<HashMap<String, SwarmMember>>>,
    pub swarms_by_id: Arc<RwLock<HashMap<String, HashSet<String>>>>,
    pub plans: Arc<RwLock<HashMap<String, VersionedPlan>>>,
    pub coordinators: Arc<RwLock<HashMap<String, String>>>,
    pub sessions: SessionAgents,
    pub soft_interrupt_queues: SessionInterruptQueues,
    pub event_history: Arc<RwLock<std::collections::VecDeque<SwarmEvent>>>,
    pub event_counter: Arc<std::sync::atomic::AtomicU64>,
    pub swarm_event_tx: broadcast::Sender<SwarmEvent>,
    pub client_event_tx: mpsc::UnboundedSender<ServerEvent>,
}
```
**Rationale:** This cuts hundreds of lines of boilerplate across the codebase, prevents accidental state desynchronization (like passing mismatched lock instances to a function), and makes testing significantly easier since you can instantiate a mock context in one line.

## 2. Prevent async awaits while holding swarm state locks
**File:** `crates/jcode-app-core/src/server/swarm.rs`
**Description:** In `salvage_assignments_of_dead_member` and `refresh_swarm_task_staleness`, the code clones the entire `SwarmState` Arcs to construct a temporary `SwarmState` just to call `persist_swarm_state_for()`. This is a band-aid for holding a write lock across an `.await` point. 
Instead, compute the mutations and collect a list of `ChangedSwarmEvent`s *inside* the lock. Drop the lock, then asynchronously persist and broadcast those events.
**Rationale:** Holding locks (even async-aware `RwLock`s) across `.await` points when those awaits trigger disk I/O creates massive contention and can freeze the server's command processing under load. 

## 3. Fix double initialization of `working_dir` in headless sessions
**File:** `crates/jcode-app-core/src/server/headless.rs`
**Description:** In `create_headless_session`, the `working_dir` is applied to the agent twice using different formatting logics:
```rust
// First time (L125):
if let Some(ref dir) = working_dir && let Some(path) = dir.to_str() {
    new_agent.set_working_dir(path);
}

// Second time (L130):
if let Some(ref dir) = working_dir {
    if let Some(dir_str) = dir.to_str() {
        new_agent.set_working_dir(dir_str); // Identical to above
    } else {
        new_agent.set_working_dir(&dir.display().to_string()); // Fallback
    }
}
```
**Rationale:** Minor cleanup, but removes confusion about which path actually takes effect and gracefully handles non-UTF8 paths correctly on the first initialization.

## 4. Optimize `O(N)` containment scans in communication fan-out
**File:** `crates/jcode-app-core/src/server/client_comm_message.rs`
**Description:** `handle_comm_message` performs `addressable_session_ids.contains(session_id)` inside a loop. Since `addressable_session_ids` is a `Vec<String>`, this is `O(N^2)` relative to swarm size. 
Change `addressable_session_ids` to a `HashSet<String>` (or reuse the existing `known_member_ids` set and intersect it logically).
**Rationale:** The swarm caps at 1000 members. A single broadcast triggers a million string comparisons in the worst case. Moving this to a `HashSet` makes the lookup `O(1)` and eliminates a potential DoS/sluggishness vector for large swarms.

## 5. Derive `SwarmRole` and `SwarmLifecycleStatus` string mappings via macros
**File:** `crates/jcode-swarm-core/src/lib.rs`
**Description:** `SwarmRole` and `SwarmLifecycleStatus` have highly repetitive 40-line `impl` blocks for `as_str`, `From<String>`, `Serialize`, and `Deserialize`. Replace these with `strum::EnumString` and `strum::Display`, or a local macro.
**Rationale:** Reduces 150 lines of pure boilerplate to a few attributes. This prevents drift when adding new lifecycle states (like `Todo` or `Pending`) and reduces compile times slightly.

## 6. Add streaming support to the MCP serve bridge
**File:** `src/cli/mcp_serve.rs`
**Description:** Currently, `handle_tools_call` executes a debug tool synchronously via `debug_command` and blocks until the tool returns the full payload. Long-running swarm tasks (or any agent execution triggered via MCP) will hang the external MCP client or cause it to timeout.
Implement a non-blocking execution wrapper: trigger the tool, stream `stdout`/tool progress events back to the MCP client (if it supports streaming, via MCP protocol 2024-11-05 `notifications/progress`), or use a heartbeat.
**Rationale:** The core value of the `mcp-serve` bridge is exposing the swarm to external editors. Without streaming, the editor UI will assume the tool died on any task taking longer than 30 seconds.

## 7. Unify global server lifecycle monitor closures
**File:** `crates/jcode-app-core/src/server/lifecycle.rs`
**Description:** `spawn_persistent_lifecycle_monitor` and `spawn_temporary_lifecycle_monitor` share almost identical interval-watching loops, `client_count` handling, and logging. 
Extract a single `monitor_server(client_count, idle_timeout_secs, orphan_check: Option<OrphanPolicy>)` function that handles the unified loop.
**Rationale:** Consolidates the daemon lifecycle logic into one easily testable point, ensuring that future hardening (like the bounded shutdown fixes) only has to be applied once.

## 8. Memoize the spawn tree for `swarm_ancestors` and relationship checks
**File:** `crates/jcode-app-core/src/server/swarm.rs`
**Description:** `swarm_is_self_or_ancestor` calls `swarm_ancestors`, which walks the `members` HashMap `O(N)` levels deep. In broadcast scopes, this is called repeatedly. 
Maintain an auxiliary `HashMap<String, HashSet<String>>` (`parent_to_children`) in `SwarmState` updated on member join/death to allow `O(1)` subtree lookups, or cache the ancestor set per session when it is spawned.
**Rationale:** Speeds up broadcast validation massively for deep DAG graphs, avoiding linear scans on every message dispatch.

## 9. Add periodic garbage collection of dead swarm members
**File:** `crates/jcode-app-core/src/server/swarm.rs`
**Description:** The `sweep_dead_pid_swarm_members` task marks dead members as `crashed` or `stopped`. However, it explicitly notes that dead members accumulate in `members` to avoid disk loads. Over days of uptime, this `HashMap` grows indefinitely.
Add a secondary sweep that purges members that have been terminal for > 1 hour, removing them from `swarms_by_id` and emitting a final `purged` event.
**Rationale:** Unbounded memory growth in the daemon's core state is a sustainability footgun for long-running sessions.

## 10. Support streaming tool output over MCP via the `tools/call` bridge
**File:** `src/cli/mcp_serve.rs`, `crates/jcode-app-core/src/server/swarm.rs`
**Description:** In `handle_tools_call` inside `mcp_serve.rs`, `self.debug_command` awaits the daemon, which awaits the entire swarm task lifecycle. 
Update the daemon's debug socket protocol to accept an optional `stream: true` parameter, pushing `ServerEvent::ToolProgress` updates over the socket as distinct JSON lines. The `mcp-serve` loop can capture these and forward them as MCP progress notifications.
**Rationale:** Brings the rich streaming UX of the native TUI to external MCP clients (like VS Code/Cursor) using the swarm.

## 11. Eliminate dead `event_txs` and unify swarm event dispatch
**File:** `crates/jcode-app-core/src/server/headless.rs`, `swarm.rs`
**Description:** When spawning a headless session, the code instantiates an empty `event_txs: HashMap::new()` inside `SwarmMember`, but manually routes notifications via `member.event_tx`. In `client_comm_message.rs`, fanouts only use `member.event_tx.send()`.
Remove the `event_txs` field completely, or document and implement what it was supposed to do (e.g., multiplexing multiple UI views of the same agent).
**Rationale:** Dead code in core state structs wastes memory (thousands of empty HashMaps) and causes confusion for future contributors.

## 12. Enforce DAG edge artifact typing at the validation layer
**File:** `crates/jcode-app-core/src/server/comm_plan.rs`
**Description:** The W1/W2 event log supports typed artifact-dataflow on graph edges, but `plan_cycle_error` only validates topological sort and cycles. 
Add a validation pass during `plan_proposal` that ensures any `PlanItem` specifying artifact outputs has corresponding inputs defined on its `depends_on` nodes, failing early if the types mismatch.
**Rationale:** Fails fast on illogical agent task delegation. Prevents a worker from spinning up, realizing its predecessor didn't generate the required file/artifact, and silently dying.

## 13. Remove legacy `OwnedChildPermit` indirection
**File:** `crates/jcode-base/src/mcp/client.rs`
**Description:** `OwnedChildPermit` is instantiated, stored in `_child_permit`, and manually attached via `attach_child_permit` to guard the spawn. It exists *only* to decrement an `AtomicUsize` on `Drop`.
Simplify this by removing the permit object entirely. Just call `try_reserve(&OWNED_MCP_CHILDREN, MAX_OWNED_MCP_CHILDREN)` before spawning, and store the boolean success/flag or use a scoped guard struct directly inside `McpClient::connect` without exposing the `attach_child_permit` API.
**Rationale:** Streamlines the MCP pooling logic and removes a clumsy "attach later" API that forces the caller to manage the client's internal memory permits.
