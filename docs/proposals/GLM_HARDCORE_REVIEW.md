## Leak / self-spawn guards

# Verdict: Conceptually sound guardrails, but the tester spawn cap is racy (TOCTOU) and the MCP self-reference guard is trivially bypassable via a shell wrapper.

## Findings

### 🔴 HIGH: TOCTOU race in tester spawn cap defeats the `MAX_TESTERS` limit
**File:** `crates/jcode-app-core/src/server/debug_testers.rs:91-92`

The `spawn_tester` function performs the cap check and the prune operation sequentially without holding any lock across both:
```rust
let live = prune_dead_testers()?;          // reads & releases file
check_tester_spawn_allowed(depth, live.len())?;  // checks against stale snapshot
// ... tokio spawn happens later, no synchronization
```
If two `tester:spawn` commands arrive concurrently, both can call `prune_dead_testers`, both see `live.len() == 7` (under the cap of 8), both pass `check_tester_spawn_allowed`, and both proceed to spawn. This allows the tester count to exceed `MAX_TESTERS`. In a pathological re-entrancy scenario (even with the depth guard, since depth only stops nesting, not parallel siblings), this can be exploited to spawn an unbounded number of tester processes.

**Failing scenario:** 9 concurrent `tester:spawn` requests -> 9 testers spawned (cap was 8).
**Fix:** Hold an async `Mutex` guard across the check-spawn-persist critical section in `spawn_tester`.

---

### 🔴 HIGH: MCP self-reference guard misses shell-wrapped invocations
**File:** `crates/jcode-base/src/mcp/protocol.rs:239-260`

The guard only matches when `command` is (or looks like) the `jcode` binary directly. Malicious or misconfigured JSON can bypass this trivially by wrapping the invocation in a shell:
```json
{
  "command": "sh",
  "args": ["-c", "jcode mcp-serve"]
}
```
The first non-flag argument here is `-c` (a flag), so the guard returns `false` immediately. Alternatively:
```json
{ "command": "bash", "args": ["-c", "exec jcode mcp-serve"] }
```
The comment explicitly claims the guard prevents "an unbounded session/process fork bomb", but an attacker (or a broken config generated by an LLM agent) who knows the daemon's behavior can route around it. This provides a false sense of security for a CRIT-class failure mode.

**Failing scenario:** User config contains `{"command": "sh", "args": ["-c", "jcode mcp-serve"]}` -> guard misses it -> daemon spawns shim -> fork bomb.
**Fix:** If blocking all shell-based re-entry is required, refuse any config where `command` is a known shell (`sh`/`bash`/`zsh`/`cmd`) and arguments contain `mcp-serve`. Alternatively, run a dry-run of the command's `PATH` resolution and argument expansion.

### 🟡 MED: Session backstop can be bypassed by lock elision / early exit
**File:** `crates/jcode-app-core/src/server/headless.rs:59-64`

```rust
{
    let current = sessions.read().await.len();
    if let Err(e) = session_backstop_allows(current) {
        crate::logging::warn(&format!("{e}"));
        return Err(e);
    }
}
```
The read lock is dropped immediately after the length check. Between the check passing and the actual `sessions.write().await.insert(...)` later in the function, an arbitrary number of other concurrent `create_headless_session` tasks can pass the check identically. If hit by `N` concurrent requests at the cap boundary, all `N` will proceed. The backstop only truly works if there's no contention.

**Failing scenario:** 100 concurrent `create_session` calls arrive when `sessions.len() == 1499` -> all pass the check -> 1599 sessions created, blowing past the 1500 backstop.
**Fix:** Upgrade to a write lock and hold it through the session insertion, or use an atomic counter (semaphore) that is only incremented on successful creation and decremented on drop.

### 🟢 LOW: Owned MCP child permit leak risk on failure
**File:** `crates/jcode-base/src/mcp/manager.rs:172-180`

```rust
let Some(permit) = OwnedChildPermit::try_acquire() else { ... };
let handle = tokio::spawn(async move {
    let result = match McpClient::connect(name.clone(), &config).await {
        Ok(mut client) => {
            client.attach_child_permit(permit);
            Ok(client)
        }
        Err(e) => Err(e),  // <-- permit dropped here
    };
    (name, result)
});
```
This is technically correct (the permit falls out of scope and decrements on `Err`), but the logic is fragile and depends on Rust's drop semantics inside `tokio::spawn` not silently holding the struct. If a future refactor stores `permit` in an intermediate variable before the match, or if the task panics before reaching the match (e.g., panic in `name.clone()`), the permit may leak until task cancellation. Consider acquiring the permit *inside* the spawned task to tightly couple its lifetime to `connect`.

### 🟢 LOW: `process_alive` semantics for tester pruning may be unreliable
**File:** `crates/jcode-app-core/src/server/debug_testers.rs:87`

```rust
t.get("pid").and_then(|p| p.as_u64())
    .map(|pid| pid != 0 && crate::server::lifecycle::process_alive(pid as u32))
```
On Unix, PIDs are recycled. If a tester dies and its PID is reused by an unrelated process (e.g., a system daemon), `process_alive` returns `true` and the dead tester remains in `testers.json`, permanently wedging one slot of the cap until the unrelated process exits. This makes `MAX_TESTERS = 8` fragile on long-running daemons.

**Fix:** Store `(pid, start_time)` and validate both, or use an active heartbeat/ping rather than just PID existence.

## Supervision / shutdown

# Verdict: Regressions found — persistent monitor now double-fires the watchdog, and short idle timeouts skip cleanup; SIGTERM fix is otherwise sound.

## Findings

### HIGH: `unregister_server_bounded` unconditionally spawns a 3s shutdown watchdog, and the caller *also* spawns one — multiple SIGTERM/idle exits stack redundant watchdog threads and mask real hangs
**File:** `crates/jcode-base/src/registry.rs:226` (combined with `server.rs:1080` and `lifecycle.rs:177`)

The diff advertises `unregister_server_bounded` as "non-blocking on shutdown," but it only adds a 2s `tokio::time::timeout` around async registry I/O — it does **not** spawn a real OS-thread watchdog. Meanwhile the SIGTERM path (`server.rs:1080`) spawns its *own* `std::thread::spawn(|| { sleep(3); exit(0); })` to guarantee exit. On the persistent idle path (`lifecycle.rs:174-178`) and the temporary shutdown path (`shutdown_temporary_server`), `unregister_server_bounded` is called **with no watchdog at all** — a stalled tokio runtime or registry lock wedges shutdown with no escape. On the SIGTERM path, if `unregister_server_bounded` itself deadlocks the runtime worker (e.g., lock contention on the registry while another task holds it across an `.await`), the 3s thread fires correctly — but the bounded wrapper gives a false sense of coverage it doesn't provide.

**Concrete scenario:** Persistent server idle for 300s → `lifecycle.rs` calls `unregister_server_bounded` → registry file lock held by another stuck task → 2s tokio timeout never elapses because the worker is starved → daemon hangs forever (no OS watchdog on this path).

**Fix:** Add an OS-thread watchdog in `spawn_persistent_lifecycle_monitor` and `shutdown_temporary_server` before calling `unregister_server_bounded`, mirroring the SIGTERM path:
```rust
std::thread::spawn(|| { std::thread::sleep(Duration::from_secs(3)); std::process::exit(EXIT_IDLE_TIMEOUT); });
crate::registry::unregister_server_bounded(&server_name).await;
std::process::exit(EXIT_IDLE_TIMEOUT);
```

---

### MED: Lost-cleanup on idle/signal paths when `IDLE_TIMEOUT_SECS < 2`
**File:** `crates/jcode-base/src/registry.rs:226-232`

`unregister_server_bounded` wraps `unregister_server` in a 2s timeout. If `IDLE_TIMEOUT_SECS` is small (e.g., in tests or a future config) and the registry I/O is slow, the timeout eats the cleanup window — but worse, the caller proceeds to `std::process::exit(EXIT_IDLE_TIMEOUT)` which terminates the process *immediately* after the bounded call returns, with no grace for any in-flight client-handling tasks to finish their work.

**Scenario:** `IDLE_TIMEOUT_SECS = 1` → monitor fires at first tick after 10s of idleness with elapsed ≥ 1 → calls bounded unregister (takes ~2s if slow) → `exit()` → any in-flight request handlers are terminated mid-write.

**Fix:** Document/enforce `IDLE_TIMEOUT_SECS >= some_minimum` (e.g., 60), or skip the idle exit if `client_count` was non-zero within the last N seconds to debounce connect races.

---

### MED: SIGTERM watchdog fires even on fast graceful exit if unregister takes 3s legitimately
**File:** `server.rs:1078-1086`

The watchdog thread sleeps a flat 3s and calls `exit(0)` unconditionally — it doesn't get cancelled when `unregister_server_bounded` returns quickly and the main path reaches `std::process::exit(0)`. That's fine for *this* process (it's already exiting), but the thread is detached and on some platforms a lingering short-sleep thread can race with cleanup flushes (logging buffers, etc.) since `exit(0)` does *not* join spawned threads. Not a hang, but the watchdog warning ("Shutdown watchdog fired; forcing exit.") will appear in logs even on clean shutdowns that take 2-3s, misattributing normal slowness to a forced exit.

**Scenario:** Slow disk, `unregister_server_bounded` takes 2.9s → main path exits with 0 → log shows "Shutdown watchdog fired" anyway because the thread wasn't cancelled.

**Fix:** Have the watchdog thread check an `AtomicBool` set by the main path on successful completion, or just accept the cosmetic log noise by documenting it.

---

### LOW: Persistent monitor starts even when this is a debug-controlled ephemeral session, defeating the original debug ergonomics
**File:** `server.rs:1521` (the `else` branch)

Previously, `debug_control_allowed()` logged "idle timeout monitor disabled" and skipped the monitor — useful for self-dev where you want the daemon to stay warm indefinitely for iteration. The diff removes that branch and unconditionally spawns the persistent monitor for any non-temporary server. Self-dev users who relied on a long-lived warm daemon now get auto-killed after `IDLE_TIMEOUT_SECS` of no clients.

**Scenario:** Developer using `JCODE_DEBUG_CONTROL=1` for a long debugging session, disconnects client to inspect state, comes back 6 minutes later → daemon has exited.

**Fix:** If the goal is purely to prevent the *orphaned* self-dev daemon case (as the comment claims), gate the monitor on `debug_control_allowed() && is_orphaned()` rather than removing the bypass entirely — or keep the bypass and add a separate orphan-specific check. The current change over-corrects.

---

## Clean
- `persistent_should_exit` unit test coverage is correct.
- `get_or_insert_with(Instant::now)` correctly preserves the first-idle timestamp across ticks (no accidental reset).
- `unregister_server_bounded` error handling (logs both `Err` and timeout) is appropriate.
- SIGTERM path no longer awaits `unregister_server` without a bound, closing the original hang vector — modulo the watchdog-cancellation cosmetic issue above.

## Swarm sweep cost

## Verdict: Inspect `member_status_is_dead` — if it's too strict or there's any non-`status`-string path back from terminal, the optimization silently leaks; the rest of the change is sound.

### Findings

**HIGH — `member_status_is_dead` may not match the exact set written by the crash reconciliation path**
`crates/jcode-app-core/src/server/swarm.rs:280` — the skip is only correct if `member_status_is_dead` returns true for *every* status string the sweep itself produces for crashed members. If the crashed branch writes e.g. `"crashed"` but `member_status_is_dead` recognizes only `"dead"`, a crashed member will be skipped forever and its record never re-reconciled.
**Repro:** a member whose pid is dead but whose `status` field was set to `"crashed"` by some other code path → `member_status_is_dead("crashed") == false` → swept every tick until it flips to whatever the helper recognizes, defeating the optimization; worse, if a later phase expected the sweep to update such a member, that update is silently skipped.
**Fix:** paste the actual body of `member_status_is_dead` (and the constants written by the crashed-pid branch) into this review so the set-equality can be verified; if they diverge, unify them via a shared `pub const DEAD_STATUSES: &[&str]`.

**MED — Mixing of "dead", "crashed", "exited", "left" as distinct terminal states risks false negatives**
`crates/jcode-app-core/src/server/swarm.rs:280` — terminal members in a swarm legitimately take multiple forms (`left`, `crashed`, `completed`, …). A helper named `*_is_dead` that only matches `["dead"]` (or `["dead", "crashed"]`) will *not* skip members in other terminal states, so the "dead members otherwise accumulate" problem the comment cites persists for those statuses — disk load stays O(all members) for any long-lived swarm with historical `"left"` members.
**Repro:** long-running swarm where members leave over time with `status == "left"` → every one of them is still `Session::load`-ed on each tick → no perf win, regression on hot-path disk I/O.
**Fix:** rename to `member_status_is_terminal` and have it match the full terminal set (whatever `member_status`'s enum-equivalent treats as terminal); update the comment accordingly.

**LOW — Comment claims proportional-to-live-members; not guaranteed if non-dead statuses exist**
`crates/jcode-app-core/src/server/swarm.rs:273` — the comment overstates the win. Only strictly correct if every non-live status is recognized as dead. Harmless as a correctness matter, but invites future reviewers to rely on the property for further refactors (e.g. "we don't need an index of live members, the sweep already does it").
**Fix:** reword to "Skip members already in a terminal status; bounded by the count of non-terminal members for the terminal set recognized by `member_status_is_dead`."

### Clean
- No regression introduced for the re-check path of genuinely live members — `!is_dead` is the right predicate direction.
- `_swarms_by_id` and `reconcile_active_sessions()` ordering unchanged.
- No new race: the read lock is held only for the snapshot, same as before; filtering inside the lock is strictly safer than outside.
- No resource leak, no auth surface touched.

## Comm migration: channels

**Verdict: CRIT — Client-to-server protocol breakage; clients sending legacy channel fields will crash the server with a JSON deserialization error.**

### CRIT

**1. `Request` enum / serde: legacy clients sending `channel` fields or `CommSubscribeChannel`/`CommListChannels` requests will fail to deserialize**
`crates/.../protocol` (not shown but referenced): The `Request::CommMessage` (now missing `channel`), `CommSubscribeChannel`, `CommUnsubscribeChannel`, `CommListChannels`, and `CommChannelMembers` variants were removed, and `NotificationType::Message` lost its `channel` field. If the `Request` / `ServerEvent` types derive `Deserialize`/`Serialize` without `#[serde(default)]` on `channel` and `#[serde(deny_unknown_fields)]` is absent (or, worse, present), this is a breaking protocol change:
- **Repro**: Old client / persisted log replays / older TUI sends `{"type":"comm_message", "channel":"foo", ...}` or `{"type":"comm_subscribe_channel", ...}`. Server fails to parse the frame and drops the connection (or returns a fatal error).
- **Fix**: Either (a) keep `channel: Option<String>` with `#[serde(default)]` on the wire structs and handle it as `None`/ignored, or (b) bump the protocol version and ensure all clients are upgraded atomically. Also verify replay of historical transcripts (which may contain `channel` in serialized `NotificationType`s) doesn't panic.

### HIGH

**2. `handle_comm_message` change to subtree broadcast fallback — legacy channel messages are now broadcast loudly instead of being dropped or scoped**
`client_comm_message.rs:229`: The old code, when `channel` was set but had no subscribers, fell back to `subtree_broadcast_targets` (a deliberate, scoped fallback). Now any message that used to carry `channel` will always hit the broadcast branch and reach the entire subtree. If any caller / client still passes a channel (see #1) or if there were semantic intents relying on channel fan-out, the blast radius of a "channel" message is silently widened from "few subscribers" to "entire spawned subtree."
- **Repro**: Coordinator sends a message intended for `#religion-debate` (2 subscribers). With removal, if the field is still passed and parsed as `None`, the message now hits every subtree member. This is an authorization/audience regression for sensitive messages.
- **Fix**: If channel semantics are truly dead, document that intent. If any client still sends `channel`, explicitly reject the request rather than silently broadcasting.

### MED

**3. `handle_comm_stop` removed channel-subscription cleanup but kept a now-unused `target_session` mutation path that may leak dangling state if channel subs are reintroduced**
`comm_session.rs:1291`: The cleanup call was removed but the surrounding control flow (event recording, agent removal) remains. If the `Server` still has any reference path that could repopulate `channel_subscriptions` (e.g., a missed migration helper, a debug endpoint, a future merge conflict resolution), the stop path will leak. Since the entire `ChannelSubscriptions` type and fields were removed from `Server`, this is likely safe, but verify there are no other producers.
- **Fix**: Confirm via grep that no code path can still mutate channel subscription state; ideally assert in a debug build that the (now-removed) maps remain empty.

### LOW

**4. Incomplete removal in `comm_session.rs` — leftover `_channel_subscriptions` parameters**
`comm_session.rs:850` (`handle_comm_spawn`): Still takes `_channel_subscriptions: &ChannelSubscriptions` and `_channel_subscriptions_by_session: &ChannelSubscriptions` (now with underscore prefix). Since `ChannelSubscriptions` type alias was removed from this file but still must resolve via imports (and the caller site was updated), these underscored params are dead weight. Also they create a maintenance trap: they imply the function still needs subscription context.
- **Fix**: Delete the two unused parameters from `handle_comm_spawn` and update call sites.

---

### Notes on clean sections
- **`client_comm_channels.rs` deletion**: Complete removal of handlers, logging, and event recording for subscribe/unsubscribe/list/members. No dead branches observed in the deleted file.
- **`client_disconnect_cleanup.rs` / `client_session.rs` removal of `remove_session_channel_subscriptions` calls**: Removed cleanly; no orphaned references in the diff.
- **`client_lifecycle.rs` request dispatch**: All four channel request arms (`CommListChannels`, `CommChannelMembers`, `CommSubscribeChannel`, `CommUnsubscribeChannel`) removed from both the heavyweight and lightweight control paths.
- **`background_tasks.rs` / `comm_control.rs` / `comm_plan.rs` / `comm_sync.rs` / `comm_session.rs` NotificationType::Message changes**: Consistent removal of `channel: None` across all DM/plan/swarm notification constructions.

**The critical risk is wire-format compatibility.** If this is a self-contained monorepo where every client and every persisted transcript is guaranteed upgraded atomically, the CRIT demotes to MED. Otherwise, clients will be disconnected on the first frame carrying a legacy field.

## Comm migration: shared-context + PlanProposalCache

### Verdict: Mostly clean migration, but contains one HIGH-severity regression (lost `created_at` semantics) and several MED/LOW-severity behavioral gaps.

### Findings

**HIGH: `created_at` reset semantics lost on re-proposal**
`crates/jcode-app-core/src/server/comm_plan.rs:217`
The old `SharedContext` preserved `created_at` if a key already existed (allowing a worker to re-propose or update without resetting the timer). The new code unconditionally overwrites with `Instant::now()`.
*Scenario:* Worker `w1` proposes a plan at `T=0`. It is rejected. `w1` proposes an updated plan at `T=100`. Previously, `created_at` remained `T=0` (so a TTL/expiry policy would still see an old proposal); now `created_at` becomes `T=100`, masking the true age of the proposal chain. Any LRU/expiry/GC based on `created_at` will be defeated.
*Fix:* Read existing `created_at` before inserting, like the old code did:
```rust
let created_at = cache.get(&swarm_id).and_then(|m| m.get(&req_session_id)).map(|p| p.created_at).unwrap_or_else(Instant::now);
```

**MED: Old proposals from former swarm members are never garbage collected**
`crates/jcode-app-core/src/server/state.rs:316` (`PlanProposalCache`)
The cache is `swarm_id -> proposer_session_id -> PendingPlanProposal`. There is no cleanup hook when a session leaves the swarm or when the swarm is destroyed. With the old shared-context store, this was still true, but now that this is the *only* load-bearing consumer, the leak is more concentrated.
*Scenario:* A coordinator repeatedly spawns transient workers that each `propose_plan` and then disconnect. Their proposals remain in the cache forever, consuming memory and showing up in `swarm:proposals` listings.
*Fix:* Add a cleanup hook (e.g., on session disconnect or swarm teardown) that removes the proposer's entry from `plan_proposals`.

**MED: Unbounded cache growth from multiple pending proposals per proposer is no longer prevented by accident**
`crates/jcode-app-core/src/server/comm_plan.rs:215`
With the KV store, `proposal_key` was `format!("plan_proposal:{req_session_id}")`, so a proposer had exactly one slot. This is preserved, but the cache now also allows other code paths to insert under arbitrary keys if the API is extended later. The type `HashMap<String, PendingPlanProposal>` doesn't enforce the one-per-proposer invariant.
*Fix:* Consider documenting the invariant or using a `struct PendingProposal` that explicitly replaces rather than merges, to prevent future regressions.

**MED: Approval path removes proposal even on cycle rejection (regression risk)**
`crates/jcode-app-core/src/server/comm_plan.rs:415` (in `handle_comm_approve_plan`)
The diff shows the cycle-rejection branch returns early *before* the `cache.write().remove()` block at line 425, which is correct (proposal stays pending). However, the *only* removal path is now inside the success branch. If a future edit adds another error path after the cycle check but before the removal, the proposal could be silently consumed. This is fragile.
*Fix:* No immediate fix needed, but worth a comment noting that early returns before line 425 intentionally keep the proposal pending.

**LOW: `SwarmEventType::ContextUpdate` removed without version bump**
`crates/jcode-app-core/src/server/state.rs:344`
If any external client subscribes to swarm events via `debug_events.rs` and filters on `"context_update"`, they will now silently never see those events. The client won't error (the match is exhaustive server-side), but the event stream is now missing a type that used to exist.
*Scenario:* A monitoring tool expecting `context_update` events sees nothing and assumes no context was shared, masking the fact that the feature was removed.
*Fix:* Document this as a breaking change in the event protocol, or keep `ContextUpdate` as a deprecated/never-emitted variant for backward compatibility.

**LOW: Debug command `swarm:context` silently returns nothing instead of erroring**
`crates/jcode-app-core/src/server/debug_help.rs:79`
The help text still mentions `swarm:context` in some places? No, it's removed. But if a user runs `swarm:context`, it now falls through to the "unknown command" path in `debug_swarm_read.rs`, returning an error. This is correct, but the error message won't explain *why* (feature removed).
*Fix:* Add an explicit check for removed commands (`swarm:context`, `swarm:set_context`) that returns a helpful "shared context has been removed; use plan proposals instead" error.

**LOW: Tool description still mentions "legacy primitive" in some branches**
`crates/jcode-app-core/src/tool/communicate.rs:1861`
The `BASE_DESCRIPTION` no longer mentions shared-context, but if `swarm_prompt` is non-empty, it's appended. If the swarm prompt still references shared-context (not in diff, so can't verify), the user-facing docs would be inconsistent.
*Fix:* Audit `load_swarm_prompt` content for shared-context references.

- **Key collision:** The migration from `plan_proposal:{session_id}` KV keys to `proposer_session_id` HashMap keys is consistent across propose/approve/reject. No collision risk.
- **Lost proposals:** Proposals are stored before fanout, and only removed on approve/reject success. No loss.

## Fleet: per-member data + cross-swarm stop + TUI view

### Verdict: Mostly clean, but the `--force` flag is dropped through the legacy agent path and a documentation mismatch in the TUI hints at an ambiguous / broken feature.

### Findings

**HIGH: Legacy / agent `stop_swarm_sessions` always drops `force: true` for cross-swarm targets**
- `crates/jcode-app-core/src/tool/communicate.rs:250-253` and `:2627-2630`
- **Scenario:** An automated agent or legacy tool calls `stop_swarm_sessions` and intentionally passes `force=true` to stop a foreign-swarm target. Because `cross_swarm` is hardcoded to `false` in this path, `resolve_stop_target_session` will refuse to find the foreign target (failing resolution entirely), and even for owned targets, the `force` flag is requested but cross-swarm permissions are not lifted. If the agent relies on automated cross-swarm cleanup (e.g., orchestrator removing zombie subagents in another swarm), the request fails or is unauthorized.
- **Fix:** Expose `cross_swarm: bool` to `stop_swarm_sessions` or explicitly document that automated cross-swarm stops are intentionally unsupported to prevent silent authorization regressions.

**MED: TUI renders cross-swarm stop hint unconditionally, but help implies it is only for unowned targets**
- `crates/jcode-tui/src/tui/app/commands_swarm.rs:349-352`
- **Scenario:** The fleet view always appends: `→ stop a member: /swarm stop <name> --cross-swarm (add --force if you don't own it)`. This trains the operator to use `--cross-swarm` blindly for *any* member, even those in their own swarm. If a coordinator shares a friendly name with a foreign swarm member, using `--cross-swarm` changes the resolution scope and widens the match, potentially causing the user to stop an unintended foreign agent if they omit `--force`.
- **Fix:** Adjust the hint to: `→ stop a member: /swarm stop <name> [--cross-swarm --force]`.

**MED: Flag parsing drops unknown flags silently if member is already parsed**
